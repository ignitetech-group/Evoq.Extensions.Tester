Code Analysis

TagManagerModule.cs Implementation

The HttpModule implements IHttpModule interface with the following key behaviors:

public class TagManagerModule : IHttpModule { public void Init(HttpApplication context) { context.PreRequestHandlerExecute += RegisterPagePrerenderHandler; } private static void RegisterPagePrerenderHandler(Object source, EventArgs e) { // Validates DNN request via Initialize.ProcessHttpModule() // Only processes Page handlers // Only injects GTM on CDefault pages (DNN default page class) } }

Key Filtering Logic:

  • Initialize.ProcessHttpModule() - Filters valid DNN requests
  • page == null check - Returns early for non-Page handlers
  • page is CDefault check - Only processes DNN's default page class

DNN Manifest Configuration

The DNNXE_GoogleTagManagerConnector.dnn manifest defines:

<!-- Install: Adds to system.webServer/modules --> <add name="GoogleTagManager" type="Evoq.GoogleTagManagerConnector.TagManagerModule, Evoq.GoogleTagManagerConnector" preCondition="managedHandler" /> <!-- Uninstall: Removes from system.web/httpModules --> <node path="/configuration/system.web/httpModules/add[@name='GoogleTagManager']" action="remove" />
Note: There's a path inconsistency between install (system.webServer/modules) and uninstall (system.web/httpModules). This may affect uninstallation in classic pipeline mode.

6. Module Removal from web.config on Uninstall (Code Review)

INFO
Analysis:
  1. Reviewed the DNN manifest uninstall configuration
  2. Identified potential path mismatch issue

Observation: The manifest installs to system.webServer/modules (IIS7+ integrated mode) but uninstalls from system.web/httpModules (classic mode). This path mismatch could cause the module to remain registered after uninstallation in integrated pipeline mode.

Recommendation: Update the uninstall configuration to remove from system.webServer/modules to match the install path, or add both paths for complete cleanup.
