Feature Description

Automatic authentication using Windows integrated authentication for domain-joined computers. This feature enables seamless single sign-on (SSO) for users on domain-joined machines, eliminating the need for manual credential entry.

Key Components

  • WindowsSignIn.aspx - The SSO login page that handles Windows credential validation
  • AuthenticationModule.cs - HTTP Module that intercepts requests and redirects to Windows authentication
  • DNNPro_ActiveDirectoryAuthentication - Authentication provider extension (version 10.1.1)

Dependencies

  • IIS Windows Authentication feature must be installed and enabled
  • Domain membership for the server and client computers
  • WindowsSignIn.aspx must have Anonymous Authentication disabled in IIS
  • Configuration sections must be unlocked at IIS level for application override

<windowsAuthentication enabled="true" /> <anonymousAuthentication enabled="false" />

Error Details:

  • HTTP Error 500.19 - Internal Server Error

  • Config Error: This configuration section cannot be used at this path. Locking is either by default (overrideModeDefault="Deny"), or set explicitly.

Prerequisites to Enable Windows SSO:
  1. Install Windows Authentication feature in IIS:
    # In PowerShell (as Administrator) Enable-WindowsOptionalFeature -Online -FeatureName "IIS-WindowsAuthentication"
  2. Unlock the authentication configuration sections in IIS:
    # In applicationHost.config or via IIS Manager <section name="windowsAuthentication" overrideModeDefault="Allow" /> <section name="anonymousAuthentication" overrideModeDefault="Allow" />
  3. Configure the site/application in IIS Manager:
    • Enable Windows Authentication
    • Disable Anonymous Authentication for WindowsSignIn.aspx
  4. Ensure the server and client computers are domain-joined

Code Analysis Summary

WindowsSignIn.aspx.cs Logic

Based on code review, the Windows SSO works as follows:

  1. When a user accesses the site, the AuthenticationModule intercepts the request
  2. If Windows Authentication is enabled in the AD configuration and the user's authentication status is undefined, they are redirected to WindowsSignIn.aspx
  3. IIS performs Windows authentication (Kerberos/NTLM)
  4. WindowsSignIn.aspx checks Request.ServerVariables["LOGON_USER"]
  5. If LOGON_USER is present, it calls AuthenticationLogon() to authenticate the user in DNN
  6. The user is then redirected back to the original page, now authenticated

AuthenticationModule.cs Features

  • Skips processing for non-Default.aspx pages and RSS feeds
  • Bypasses authentication for already logged-in Admin/Host users
  • Supports IP range filtering for selective auto-login (AutoIP configuration)
  • Manages authentication status cookies (DNNReturnTo, authentication status)
  • Handles bot/crawler detection to prevent unnecessary authentication prompts

Recommendations

To Enable Windows SSO Testing

  1. Configure IIS at server level - Enable Windows Authentication and unlock configuration sections
  2. Test on a domain-joined machine - Windows SSO requires Active Directory domain membership
  3. Configure the AD Provider settings - Enable the authentication provider and configure LDAP connection settings
  4. Verify network connectivity - Ensure the web server can communicate with Active Directory domain controllers

Production Deployment Considerations

  • Consider using IP range filtering to limit auto-SSO to internal network users
  • Configure appropriate bot/crawler exclusions to prevent authentication loops
  • Test the logout and re-authentication flow to ensure proper session management
  • Document the IIS configuration requirements for deployment teams
