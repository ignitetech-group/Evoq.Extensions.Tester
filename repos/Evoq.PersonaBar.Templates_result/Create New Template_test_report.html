Code Review Finding

Code analysis shows script injection validation is implemented using regex pattern in templatesCreateManager.js:

ko.validation.rules.pattern.hasScript = /<[\/]?[ ]{0,}[\t]{0,}script\b/gi;

ko.validation.rules['hasScript'] = {
    validator: function (val) {
        if (val) {
            return String(val).match(ko.validation.rules.pattern.hasScript) == null;
        }
        return true;
    },
    message: ''
};

    viewModel.pageTemplates().forEach(function (t) { t.selected(false); });

    viewModel.visibleAddTemplateForm(false);

Observations

  • Server API Issues: The Templates API endpoints are returning HTTP 500 errors, which blocked comprehensive testing of the Create New Template feature. This appears to be a server-side configuration or code issue.
  • Template Prerequisite Design: The Create New Template feature requires at least one existing template to serve as a "layout" for new templates. This design decision means templates cannot be created from scratch - only from existing templates.
  • Code Validation Present: Code review confirms that client-side validation is implemented for:
    • Required template name field
    • Script injection prevention (XSS protection)
    • Template selection requirement
  • UI Accessibility: The feature UI is accessible and navigation works correctly. The "Add Template" button and form elements are properly rendered.
  • Error Handling: The system displays appropriate error messages when API calls fail and shows helpful guidance when prerequisites are not met.
  • Relevant Files:
    • Services/TemplatesController.cs - Backend API controller
    • admin/personaBar/scripts/templatesCreateManager.js - Frontend validation and logic
    • admin/personaBar/Templates.html - UI template
  • Recommendation: Investigate the server-side 500 errors before re-running the full test suite. The errors appear to originate from the GetPageTemplates and SavePageDetails API endpoints.
