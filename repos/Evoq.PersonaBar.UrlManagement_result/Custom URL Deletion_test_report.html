Custom URL Deletion - Test Report

Extension: Evoq.PersonaBar.UrlManagement (PersonaBar Module)

Feature: Custom URL Deletion

Description: Delete existing custom URLs from the system

Priority: High

UI Location: Admin > Persona Bar > Pages > [Page] > Advanced > S.E.O. > URLs for this page

Test Date: January 6, 2026

Test Results

Test 1: Delete Existing Custom URL Successfully PASS

Objective: Verify that a custom URL can be deleted successfully

  1. Navigate to Pages in Persona Bar
  2. Select "URL Test Page" from the page list
  3. Go to Advanced tab > S.E.O. sub-tab
  4. Locate the "URLs for this page" section with custom URLs
  5. Click the delete icon (first icon) on "/test-delete-url-1"
  6. Confirmation dialog appears: "Are you sure you want to remove this URL?"
  7. Click "Delete" to confirm
  8. URL is removed from the list

Result: The custom URL "/test-delete-url-1" was successfully deleted. The confirmation dialog appeared as expected, and after confirming, the URL was removed from the list immediately.

Test 2: Delete URL and Verify Removal from Listings PASS

Objective: Verify that deleted URLs are properly removed from the URL listings

Test Summary

Passed: 3
Failed: 0

Overall Result: All tests PASSED. The Custom URL Deletion feature works as expected.

Key Findings:

  • Custom URLs can be deleted successfully through the UI
  • A confirmation dialog is shown before deletion, preventing accidental removals
  • Deleted URLs are immediately removed from the list
  • Multiple URLs can be deleted in sequence without issues
  • The system-generated "Automatic" URLs (like /URL-Test-Page) cannot be deleted (no delete icon shown), which is correct behavior

Observations

  • UI Icon Design: The delete and edit icons may be counterintuitive - the first icon (which looks like a pencil) triggers deletion, while the second icon (which looks like a trash can) triggers editing. This could be a UX improvement opportunity.
  • URL Management Location: The URL management is per-page (under Pages > [Page] > Advanced > S.E.O.) rather than in a centralized location. This is by design and makes sense for page-specific custom URLs.
  • Non-existent URL Test: Could not test "attempt to delete non-existent URL" as this scenario requires API-level testing - the UI only shows existing URLs and there's no way to attempt deletion of a URL that doesn't exist through the UI.
  • Permission Test: Deletion permissions were not explicitly tested with different user roles as this requires separate user accounts. The superuser (host) has full deletion permissions as expected.
  • Related Data Cleanup: The code review shows that TabController.Instance.DeleteTabUrl() handles the cleanup, but database-level verification was not performed as part of UI testing.
