Observations

  • IndexThreads Setting Not Exposed in UI: The code reveals that each SearchCrawler task has an "IndexThreads" setting (SearchSpiderHelper.cs:588) that controls the number of concurrent worker threads used for crawling. However, this setting is NOT exposed in the standard Scheduler task editor UI. The default value is 1 thread. To configure a different thread count would require direct database modification or API access to the schedule settings.
  • System-Level vs Task-Level Threading: The "Max Threads" setting visible in the Scheduler header is a system-level configuration for the entire DNN Scheduler, not specific to the SearchCrawler. The per-task IndexThreads setting for crawling operations is separate and configured differently.
  • Thread Safety Implementation: Code review shows proper thread safety mechanisms including:
    • Lock statements for synchronization (Spider.cs:213, 314)
    • Monitor.Wait/Pulse for thread coordination (Spider.cs:318, 252)
    • Thread-safe Hashtable for URL tracking
  • DocumentWorker Pattern: The Spider.cs creates multiple DocumentWorker instances (line 357-360) based on the thread count parameter, allowing parallel processing of URLs during crawling operations.
