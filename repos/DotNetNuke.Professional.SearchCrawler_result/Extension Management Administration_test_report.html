2. Host Whitelist Enforcement Pass

Steps:
  1. Click "Add File Type" for Included Extensions
  2. Enter "rtf" (not in host whitelist)
  3. Click Save
  4. Observe error indicator
  5. Repeat with "jpg" and "xml"

Result: All three extensions (rtf, jpg, xml) were rejected with error indicators, demonstrating that the host whitelist enforcement is working correctly.

Code Implementation: The AddExtensionToPortalIncludedExtensionList method in SearchSpiderHelper.cs checks against GetHostAllowedExtensionWhitelist() and returns AddExtensionResult.NotInWhitelist if the extension is not allowed.

5. Duplicate Extension Prevention Pass

Steps:
  1. Click "Add File Type" for Included Extensions

Result: The system correctly rejected the duplicate entry with an error indicator, preventing duplicate extensions from being added.

Code Implementation: The AddExtensionToPortalIncludedExtensionList method checks existingExtensions.All(x => x.Trim().ToLowerInvariant() != formatedExtension) and returns AddExtensionResult.Duplicate if the extension already exists.

Key Findings

  • Whitelist Enforcement: The system correctly enforces the host file extension whitelist, rejecting extensions that are not in the allowed list.
  • CRUD Operations: Both add (with validation) and remove operations work correctly for included and excluded extensions.
  • Duplicate Prevention: The system prevents duplicate extensions from being added.
  • IFilter Integration: IFilter support information is correctly displayed, showing which extensions have content crawling capabilities.
  • Portal-Specific Settings: The system supports portal-specific extension settings as confirmed by code review.

Observations

  • The host whitelist on this test environment is restrictive, limiting which extensions can be added.
  • Some newer Office formats (.pptx, .xlsx) show "Content crawling is unavailable" due to missing IFilter support on the server.
  • The UI provides clear visual feedback (error indicators) when operations fail.
